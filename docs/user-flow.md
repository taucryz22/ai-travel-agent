# User Flow

## Цель сценария

Пользователь хочет быстро получить персональный туристический маршрут по естественному текстовому запросу.

---

## Основной пользовательский сценарий

### Шаг 1. Ввод запроса
Пользователь открывает приложение и вводит запрос в свободной форме.

Примеры:
- «Я уже в Питере и хочу рок-бары и современное искусство»
- «Хочу поехать из Москвы в Саратов на 3 дня. Хочу увидеть достопримечательности, вкусно поесть. Придумай культурную программу.»

Также пользователь указывает:
- количество дней
- бюджет
- режим перемещения

---

### Шаг 2. Отправка запроса
Frontend отправляет данные на backend через `POST /api/plan`.

---

### Шаг 3. Разбор текста
Backend пытается понять:
- город отправления
- город назначения
- город маршрута
- интересы
- темп поездки
- тип атмосферы

Если локальная LLM доступна — используется она.  
Если нет — применяется fallback-логика и дополнительные проверки через OSM.

---

### Шаг 4. Уточнение города
Если город не удалось определить однозначно:
- backend пытается выделить кандидатов из текста,
- валидирует их через Nominatim,
- выбирает подходящий город маршрута.

---

### Шаг 5. Получение смысловых подсказок
По выбранному городу подтягиваются текстовые данные из Wikivoyage.  
Из них извлекаются полезные фрагменты для контекста маршрута.

---

### Шаг 6. Генерация поисковых фраз
На основе:
- города
- интересов
- подсказок из Wikivoyage

формируются поисковые фразы для поиска мест.

Примеры:
- достопримечательности
- музей
- театр
- ресторан локальная кухня
- кафе
- рок-бар

---

### Шаг 7. Поиск мест
Через OSM/Nominatim backend получает реальные точки:
- музеи
- рестораны
- бары
- парки
- достопримечательности
- другие места

---

### Шаг 8. Построение маршрута
Планировщик:
- группирует места по категориям
- распределяет их по слотам дня
- оценивает время и расстояние между точками
- выбирает способ перемещения
- контролирует бюджет
- формирует итоговый маршрут

---

### Шаг 9. Показ результата
Frontend показывает:
- город
- маршрут по дням
- отдельные точки
- время на переход
- расстояние
- способ перемещения
- стоимость
- сводку по бюджету

---

### Шаг 10. Экспорт
Пользователь может:
- скопировать маршрут
- скачать маршрут в `.txt`
- скачать маршрут в `.pdf`

---

## Дополнительные сценарии

### Перегенерация дня
Пользователь может перегенерировать конкретный день, исключив уже выбранные точки.

### Замена точки
Пользователь может заменить отдельную точку маршрута на альтернативную.

### Переход в карты
Пользователь может нажать на ссылку и открыть маршрут в Яндекс.Картах.

---

## Логика выбора транспорта

### Пешком
Выбирается, если:
- расстояние небольшое
- переход реалистичен
- пользователь любит прогулки

### Общественный транспорт
Выбирается, если:
- расстояние среднее
- пешком идти уже невыгодно
- такси не даёт достаточного преимущества

### Такси
Выбирается, если:
- расстояние значительное
- бюджет позволяет
- это ощутимо экономит время

При такси дополнительно закладывается буфер времени на вызов и ожидание.

---

## Ограничение на прогулки
Даже если пользователь любит прогулки:
- система не должна отправлять его слишком далеко пешком
- есть мягкий и жёсткий лимит пешей дистанции

---

## Итоговый UX
С точки зрения пользователя приложение работает как интеллектуальный помощник:
1. пользователь пишет запрос обычным языком,
2. система понимает намерение,
3. находит реальные места,
4. собирает маршрут,
5. показывает удобный план поездки.
